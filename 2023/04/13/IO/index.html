IO
2023-04-13 14:22:52 #Java

输入类型的IO调用

  1. 进程向操作系统请求外部数据
  2. 操作系统将外部数据加载到内核缓冲区
  3. 操作系统将数据从内核缓冲区拷贝到进程缓冲区
  4. 进程读取数据继续后面的工作

IO步骤

  1. IO调用 - 进程发起
  2. IO执行 - 操作系统执行
    • 准备数据阶段 - 内核等待IO设备准备好数据
    • 拷贝数据阶段 - 将数据从内核缓冲区拷贝到用户空间缓冲区

同步和异步IO

  1. 描述的是针对当前执行线程/进程，发起IO调用后，当前线程/进程是否挂起等待操作系统的IO执行完成
  2. 同步
    • 程序发起IO调用后，当前线程或进程需要等待操作系统完成IO工作并告知进程已经完成，线程/进程才能继续向下执行指令
    • 请求方从发起请求到数据最后完成的过程都需要自己参与
  3. 异步
    • 程序发起IO调用后，不必等待操作系统IO执行完毕，可以向下继续执行指令，操作系统完成IO后，当前进程/线程会收到操作系统的通知
    • 应用发送完指令后就不再参与过程，只需等待最终完成结果的通知

阻塞&非阻塞

  1. 阻塞IO
    • 内核IO 操作彻底完成后，才返回用户空间，执行用户的操作
    • 传统的IO模型都是同步阻塞IO
    • Java中默认创建的socket是阻塞的
  2. 非阻塞IO
    • 用户空间拿到状态就返回
    • 非阻塞IO要求socket被设置为NONBLOCK

BIO

  1. 在内核进行IO执行的两个阶段，用户线程都被block了
  2. 程序简单，阻塞等待数据期间，用户线程挂起，基本不会占用CPU资源

Non-Blocking IO （同步非阻塞

  1. 相对于BIO提升了性能
  2. 频繁的轮询导致频繁的系统调用，会耗费大量的CPU资源
  3. 频繁轮询导致的无效系统调用

IO多路复用（同步非阻塞

  1. 由一个线程监控多个网络请求
  2. Java中的New IO
    • Channel
    • Selector
      • 多路复用器
      • 只需要一个线程便可以管理多个客户端连接，当客户端数据到了才会为其服务
    • Buffer
    • IO多路复用模型中，线程首先发起 select 调用，询问内核数据是否准备就绪，等内核把数据准备好了，用户线程再发起 read 调用，read 调用的过程还是阻塞的
  3. 目前支持IO多路复用的系统调用
    • select
      • 几乎在所有的操作系统上都有支持
      • 支持一次查询多个系统调用的可用状态
    • epoll
      • linux2.6版本
      • 属于select 调用的增强版本，优化了IO执行效率
  4. 可以同时处理成千上万个连接，系统不必创建线程/维护线程，减小系统开销

信号驱动IO模型（同步非阻塞

  1. 由于大部分情况下的轮询都是无效的，所以考虑使用信号驱动IO模型
  2. 在调用sigaction时建立一个SIGIO的信号联系，当内核数据准备好之后再通过SIGIO信号通知线程数据准备好后的可读状态，线程收到通知后再发起读取数据的请求
  3. 由于信号驱动IO的模型下应用线程在发出信号监控后即可返回，不会阻塞，所以一个应用线程也可以同时监控多个网络请求

异步IO（异步非阻塞

  1. Java中的AIO
  2. 应用只需要向内核发送一个read请求，告诉内核它要读取数据，然后立刻返回
  3. 内核收到请求后会建立一个信号联系，当数据准备就绪，内核会主动把数据从内核复制到用户空间，等所有操作完成后，内核会发起一个通知告诉应用
  4. 需要完成事件的注册与传递，需要底层操作系统提供大量支持

基础知识

  1. read 系统调用
    • 把数据从内核缓冲区复制到进程缓冲区
  2. write 系统调用
    • 把数据从进程缓冲区复制到内核缓冲区
  3. 内核缓冲区和磁盘的读写
    • 由操作系统内核完成

缓冲区

  1. 为了减少频繁的系统IO调用
